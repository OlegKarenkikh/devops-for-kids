### 1. Метафоры и аналогии — на высшем уровне

Курс блистательно использует аналогии из реальной жизни. Это лучшее, что в нём есть:

- **Docker = ланч-бокс** 🍱 — «Ты кладёшь в него еду дома, и она приезжает в офис точно такой же» — мгновенно понятно!

- **Docker Compose = целый город сервисов** 🏙️ — «Один файл — один запуск» — отражает суть

```markdown
## CI/CD: автоматизация сборки и деплоя

CI/CD = Continuous Integration / Continuous Deployment.

Аналогия: представь, что каждый раз когда ты сохраняешь документ Word,
компьютер автоматически:
1. Проверяет орфографию (CI — тесты)
2. Распечатывает и отправляет его нужным людям (CD — деплой)
```

```yaml
name: CI Pipeline

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v6

    - name: Установить Python
      uses: actions/setup-python@v6
      with:
        python-version: '3.11'

    - name: Установить зависимости
      run: pip install -r requirements.txt

    - name: Запустить тесты
      run: python -m pytest

    - name: Собрать Docker образ
      run: docker build -t my-app:${{ github.sha }} .
```

#### Проблема 6: Урок 7 (Bash-скрипты) слишком поверхностный

Для junior DevOps-инженера bash-скрипты — это ежедневный инструмент. Без условий и циклов скрипты бесполезны.

```bash
#!/bin/bash

if docker ps > /dev/null 2>&1; then
    echo "✅ Docker работает"
else
    echo "❌ Docker не запущен. Запусти: sudo systemctl start docker"
    exit 1  # Прерываем скрипт с ошибкой
fi
```

**Циклы (for):**
```bash
#!/bin/bash
# Проверить статус нескольких сервисов

SERVICES=("nginx" "docker" "ssh")

for service in "${SERVICES[@]}"; do
    if systemctl is-active --quiet "$service"; then
        echo "✅ $service — работает"
    else
        echo "❌ $service — остановлен"
    fi
done
```

**Функции:**
```bash
#!/bin/bash
# Скрипт деплоя с логированием

log() {
    echo "[$(date '+%H:%M:%S')] $1"
}

deploy_app() {
    local image=$1
    log "Начинаю деплой: $image"
    docker pull "$image" && log "Образ загружен"
    docker stop my-app 2>/dev/null || true
    docker run -d --name my-app -p 8080:8080 "$image"
    log "✅ Деплой завершён!"
}

deploy_app "my-app:latest"
```

---

## Python для DevOps: минимальный необходимый минимум

Ты не становишься Python-разработчиком. Но понять 5 концепций нужно:

### 1. Функции
```python
# Функция = действие с именем. def = define (определить)
def поздороваться(имя):
    print(f"Привет, {имя}!")

поздороваться("Вася")  # → Привет, Вася!
```

### 2. Словари (dict) — это JSON в Python
```python
# Словарь = пары ключ:значение (как JSON объект)
предмет = {
    "id": 1,
    "name": "яблоко",
    "emoji": "🍎"
}
print(предмет["name"])  # → яблоко
```

### 3. Списки (list)
```python
# Список = массив объектов
items = [
    {"id": 1, "name": "яблоко"},
    {"id": 2, "name": "банан"}
]
print(len(items))  # → 2 (количество элементов)
```

### 4. Декоратор @app.route — адрес URL
```python
# @app.route("/items") = "когда приходит запрос на /items — вызови эту функцию"
@app.route("/items", methods=["GET"])
def get_all():
    return jsonify(items)
# Это НЕ магия — это просто Python decorator (обёртка над функцией)
```

### 5. import — подключение библиотек
```python
# import = как "подключить дополнение"
from flask import Flask, jsonify, request
# flask — название библиотеки
# Flask, jsonify, request — конкретные инструменты из неё
```

#### Рекомендация 4: Добавить «Типичные ошибки» в каждый модуль

В модуле 0 и Docker есть таблицы ошибок. Это очень полезно. Нужно добавить во все модули. Примеры для модулей которым не хватает:

**Git (Модуль 2) — типичные ошибки:**
```markdown
| Ошибка | Причина | Решение |
|--------|---------|---------|
| `Please tell me who you are` | Не настроен email/name | `git config --global user.email "ты@email.com"` |
| `rejected — non-fast-forward` | На GitHub есть изменений которых нет локально | `git pull --rebase origin main` затем снова `git push` |
| `CONFLICT (content): Merge conflict in file.py` | Два человека изменили одну строку | Открыть файл, найти `<<<<<<`, выбрать нужную версию, сохранить |
| `Permission denied (publickey)` | SSH-ключ не добавлен в GitHub | Добавь ключ: GitHub → Settings → SSH Keys |
| `nothing to commit, working tree clean` | Нет изменений | Это НЕ ошибка! Измени файл и снова `git add` |
```

**Kubernetes (Модуль 5) — типичные ошибки:**
```markdown
| Статус | Причина | Диагностика |
|--------|---------|------------|
| `CrashLoopBackOff` | Приложение падает при старте | `kubectl logs pod-name --previous` |
| `ImagePullBackOff` | Образ не найден в реестре | Проверь имя образа: `kubectl describe pod pod-name` |
| `Pending` | Нет ресурсов или нет подходящей ноды | `kubectl describe pod pod-name` → раздел Events |
| `OOMKilled` | Превышен лимит памяти | Увеличь `resources.limits.memory` в Deployment |
| `Error: ImagePullBackOff` | Неправильный тег образа | Проверь что образ существует: `docker images` |
```

#### Рекомендация 5: Интерактивные упражнения с автопроверкой

**Проблема:** Задания в курсе есть, но нет механизма проверки — студент не знает правильно ли он выполнил задание.

**Предложение:** Добавить скрипты автопроверки для ключевых заданий:

```bash
#!/bin/bash
# check_module1.sh — Автопроверка Модуля 1

PASS=0
FAIL=0

check() {
    local desc=$1
    local result=$2
    if [ "$result" = "true" ]; then
        echo "✅ $desc"
        ((PASS++))
    else
        echo "❌ $desc"
        ((FAIL++))
    fi
}

# Проверка 1: Домашняя папка существует
check "Домашняя папка ~/my-course существует" \
    "$([ -d ~/my-course ] && echo true || echo false)"

# Проверка 2: hello.txt создан
check "Файл ~/my-course/hello.txt существует" \
    "$([ -f ~/my-course/hello.txt ] && echo true || echo false)"

# Проверка 3: SSH-ключ создан
check "SSH-ключ ~/.ssh/id_ed25519 существует" \
    "$([ -f ~/.ssh/id_ed25519 ] && echo true || echo false)"

echo ""
echo "Результат: $PASS пройдено, $FAIL не пройдено"
[ $FAIL -eq 0 ] && echo "🎉 Отлично! Переходи к Модулю 2" || echo "⚠️ Доделай задания перед переходом"
```

#### Рекомендация 6: Добавить секцию про работу с облаками

**Проблема:** Junior DevOps сегодня почти всегда работает с облачными провайдерами (AWS, GCP, Azure). В курсе об этом ни слова.

**Рекомендация:** Добавить хотя бы вводный урок (возможно, в бонусном модуле).

Содержание такого урока:

## Облака: AWS, GCP, Azure — что это и зачем?

Ты построил сервис на своём компьютере. Хорошо. Но как дать доступ
пользователям из интернета?

Нужен сервер в интернете. Можно купить физический (дорого, сложно).
Или взять в аренду виртуальный у облачного провайдера.

Три главных провайдера:
- **AWS** (Amazon Web Services) — самый большой, 30%+ рынка
- **GCP** (Google Cloud Platform) — сильный в ИИ и данных
- **Azure** (Microsoft) — популярен в корпоративном секторе

Бесплатные уровни для обучения:
- AWS Free Tier: 12 месяцев бесплатно для новых аккаунтов
- GCP Free Tier: $300 кредитов на 90 дней
- Azure Free: $200 кредитов на 30 дней

Минимальный пример — **запустить наш Flask API в AWS** лучше строить не на «магической» команде, а на официальном туториале, чтобы:
- использовать **актуальные AMI для выбранного региона**;
- избежать ошибок с устаревшими ID образов;
- сразу привыкать к **безопасной работе с облаком**.

Предлагаем вместо готовой команды дать ученику такой маршрут:

1. Создать аккаунт AWS и внимательно прочитать условия **AWS Free Tier**.
2. Пройти официальный гайд по запуску приложения на EC2/через консоль:
   - https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/EC2_GetStarted.html
   — или пошаговый туториал из раздела Tutorials в AWS Console (Launch a virtual machine).
3. При необходимости установить AWS CLI по официальной инструкции:
   - https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html
4. Для развёртывания своего Flask API использовать:
   - либо user data (скрипт инициализации) из официальных примеров;
   - либо готовые инструкции, где AMI подбирается автоматически для нужного региона.

> ⚠️ **Важно добавить в курс явное предупреждение:**
> - Все облачные ресурсы (EC2, базы данных и т.п.) **могут стоить денег**, даже на Free Tier.
> - После экспериментов **обязательно удаляй инстансы и другие ресурсы**, иначе счёт может расти.
> - Не храни AWS access keys в Git-репозиториях и общих файлах.
> - Создавай учётки/роли с **минимально необходимыми правами** (principle of least privilege).
> - По возможности используй **SSO/role** вместо «вечных» access keys на локальной машине.

#### Рекомендация 7: Глоссарий всех терминов курса

**Проблема:** В модуле 0 есть объяснения терминов, но они распределены по всему тексту. Новичку нужен **единый глоссарий** куда можно быстро вернуться.

```markdown
## Глоссарий: DevOps терминология для новичков

| Термин | Простыми словами | Пример использования |
|--------|-----------------|---------------------|
| **Сервер** | Компьютер который работает для других 24/7 | Сервер ВКонтакте хранит твои фото |
| **Терминал** | Способ говорить с компьютером текстом | `ls`, `mkdir`, `docker run` |
| **SSH** | Безопасный вход на удалённый сервер | `ssh user@192.168.1.1` |
| **Git** | Система версий кода (как «Отменить» для всего проекта) | `git commit`, `git push` |
| **GitHub** | Облачное хранилище для Git репозиториев | github.com |
| **Docker** | Упаковка приложения со всеми зависимостями | `docker run my-app` |
| **Контейнер** | Запущенный Docker образ (как запущенная программа) | `docker ps` показывает контейнеры |
| **Образ (Image)** | Шаблон для контейнера (как установочный файл) | `docker images` показывает образы |
| **Dockerfile** | Рецепт сборки Docker образа | `FROM python:3.11 \n CMD python app.py` |
| **Kubernetes (K8s)** | Система управления множеством контейнеров | Запускает и следит за 100 контейнерами |
| **Pod** | Наименьшая единица в Kubernetes (1+ контейнеров) | `kubectl get pods` |
| **Deployment** | Инструкция K8s: сколько Pod'ов держать | `replicas: 3` — всегда 3 копии |
| **Service** | Стабильный адрес для группы Pod'ов | Pod'ы перезапускаются, Service — нет |
| **YAML** | Формат конфигурационных файлов (отступы важны!) | `docker-compose.yml`, `deployment.yaml` |
| **REST API** | Интерфейс для общения программ по HTTP | `GET /items`, `POST /items` |
| **CI/CD** | Автоматическая проверка и деплой кода | GitHub Actions, Jenkins |
| **Environment Variable** | Переменная вне кода (пароли, настройки) | `DB_PASSWORD=secret` в `.env` |
| **.env файл** | Файл с переменными окружения (НЕ в Git!) | `DB_PASSWORD=Мой_Секрет_123` |
| **Port** | Номер «квартиры» на сервере | `:8080`, `:443` |
| **Volume** | Папка вне контейнера для хранения данных | `docker run -v /data:/app/data` |
| **Replica** | Копия сервиса для надёжности | K8s запускает 3 реплики = 3 копии |
| **Load Balancer** | Распределяет запросы между репликами | Если реплика упала — другие продолжают работать |
| **Healthcheck** | Проверка: работает ли сервис? | K8s убивает Pod если healthcheck не проходит |
```
