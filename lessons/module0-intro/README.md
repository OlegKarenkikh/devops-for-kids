## Что такое терминал — и зачем он вообще нужен

Представь, что компьютер — это очень послушный помощник. Обычно ты общаешься с ним **мышкой**: нажимаешь кнопки, иконки, меню. Это удобно, но медленно и ограниченно.

Терминал — это способ говорить с компьютером **текстом**. Ты пишешь команду — компьютер немедленно выполняет. Это как разница между SMS и живым разговором: текстом медленнее набираешь, зато можешь сказать точно что нужно.

**Зачем нужен терминал в нашем курсе:**
- Серверы в интернете не имеют экрана, мышки и кнопок — только терминал
- Docker, Git, Kubernetes управляются только через команды
- Одной командой делаешь то, на что мышкой ушли бы часы
- Можно автоматизировать — написать скрипт, который делает всё сам

> **Важно понять:** профессиональный разработчик или DevOps-инженер проводит в терминале бо́льшую часть рабочего времени. Это не страшно — это просто навык, который приходит с практикой.

## Что ты видишь — строка приглашения

После открытия терминала ты увидишь что-то похожее на:

```
user@my-computer:~$
```

Это **строка приглашения** — компьютер говорит тебе «я готов, давай команду». Разберём каждый символ:

```
user            ← Твоё имя пользователя (кто ты)
@               ← Символ "на" (разделитель)
my-computer     ← Имя твоего компьютера (где ты)
:               ← Разделитель
~               ← Ты сейчас в домашней папке (тильда = дом)
$               ← Приглашение: "я готов, пиши команду"
```

**Зачем знать каждую часть?** Когда ты подключишься к серверу в интернете, строка будет другой:
```
root@server-42:/var/www$
```
И ты сразу поймёшь: `root` — я администратор, `server-42` — это сервер в интернете, `/var/www` — я в папке с сайтом.

> **Важно:** В примерах этого курса `$` показывает начало строки-команды — **его вводить не нужно!** Это просто обозначение.

## Твоя первая команда — абсолютно безопасная

```bash
# Напечатай это и нажми Enter:
whoami
```

Компьютер ответит твоим именем пользователя. Ничего не изменилось, ничего не сломалось — команда только прочитала информацию.

**Что именно произошло — пошагово:**
1. Ты напечатал `whoami` (буквально: «кто я?»)
2. Нажал Enter — отправил запрос
3. Компьютер нашёл программу с именем `whoami`
4. Запустил её
5. Она прочитала имя текущего пользователя из системы
6. Напечатала его на экране
7. Снова появился `$` — компьютер ждёт следующей команды

**Зачем понимать эти 7 шагов?** Когда придёт ошибка `whoami: command not found`, ты будешь знать что пошло не так — на шаге 3 компьютер не нашёл программу.

> **🎯 Задание:** Введи `whoami`, потом `pwd` (показывает где ты), потом `date` (текущая дата). После каждой команды — пауза. Подумай что произошло.

## Анатомия команды — как читать любую команду

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/command_anatomy.png" alt="Анатомия команды терминала" width="90%"/>
<br/><em>Каждое слово в команде что-то значит. Научись читать команды — не бойся их.</em>
</div>

Команда в терминале — это **предложение на языке компьютера**. У неё такая же структура как у русского предложения:

```
ГЛАГОЛ  + КАК (наречие)   + ЧТО (дополнение)
ls        -la
docker    run -d -p 8080   my-app
git       add               .
cp        -r                папка/ /новое-место/
```

### Что такое флаги и зачем они нужны

Флаги (начинаются с `-` или `--`) — это **уточнения** к действию. Как наречия в русском языке: «бежать» и «бежать быстро» — разные вещи.

| Короткий флаг | Длинный флаг | Что значит |
|--------------|-------------|-----------|
| `-d` | `--detach` | В фоне, не занимай терминал |
| `-p 8080:8080` | `--publish 8080:8080` | Открой порт хост:контейнер |
| `-a` | `--all` | Показать всё (включая скрытое) |
| `-l` | `--long` | Подробный формат |
| `-v` | `--volume` | Подключить папку |
| `-r` | `--recursive` | Рекурсивно (для папок) |

> **Лайфхак для новичков:** всегда можно написать длинную версию — она понятнее:
> `docker run --detach --publish 8080:8080 my-app`
> Это та же команда что `docker run -d -p 8080:8080 my-app`. Результат одинаковый.

> **🎯 Задание:** Посмотри на команду `ls -la`. Переведи её в длинную форму: `ls --all --long`. Запусти обе — убедись что результат одинаковый.

## Как читать ответы терминала — зелёный и красный

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/terminal_output.png" alt="Успех и ошибка в терминале" width="90%"/>
<br/><em>Ошибка — это подсказка, не катастрофа. Главное — прочитать её до конца.</em>
</div>

### Когда всё хорошо

Обычно терминал **молчит** или пишет немного текста, затем снова показывает `$`. Тишина = успех.

```bash
mkdir my-project    # Создать папку
                    # ← Нет ответа = всё хорошо, папка создана
$                   # ← Снова ждёт команду
```

### Когда что-то пошло не так

```bash
cat несуществующий.txt
# Ответ:
cat: несуществующий.txt: No such file or directory
```

**Четыре правила при любой ошибке:**

1. **Не паникуй.** Просто увидеть ошибку — не сломало компьютер. Ничего серьёзного не случилось.
2. **Прочитай последнюю строку до конца.** Там написано ЧТО именно пошло не так.
3. **Найди ключевые слова** — по ним поймёшь причину:

| Текст ошибки | Причина | Что делать |
|-------------|---------|-----------|
| `No such file or directory` | Файл или папка не существует | Проверь путь командой `ls` |
| `Permission denied` | Нет прав на это действие | Добавь `sudo` перед командой |
| `command not found` | Программа не установлена | Установи: `sudo apt install имя` |
| `Address already in use` | Порт занят другой программой | Смени порт или останови другую |
| `syntax error` | Опечатка в команде | Найди лишний/пропущенный символ |

4. **Скопируй точный текст ошибки** → поищи в Google → так делают все программисты мира.

## Точка, две точки и тильда — невидимые ориентиры

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/path_symbols.png" alt="Спецсимволы в путях Linux" width="90%"/>
<br/><em>Три символа которые встречаются везде — запомни раз и навсегда.</em>
</div>

```bash
.    # Точка = ЗДЕСЬ (текущая папка где я сейчас)
..   # Две точки = ВЫШЕ (папка на уровень вверх)
~    # Тильда = ДОМ (домашняя папка пользователя)
```

**Аналогия с Проводником Windows или Finder на Mac:** когда ты открываешь папку — ты видишь её содержимое (это `.`). Кнопка «Назад» — это `..`. Кнопка «Домашняя папка» — это `~`.

**Как это используется в реальных командах:**

```bash
git add .          # Добавить ВСЁ что есть в текущей папке
docker build .     # Собрать образ из текущей папки (Dockerfile здесь)
cd ..              # Перейти в папку выше (как кнопка "Назад")
cd ~               # Вернуться домой (работает откуда угодно)
cp файл.txt ..     # Скопировать файл в папку выше
ls ..              # Посмотреть содержимое папки выше
```

> **🎯 Упражнение — выполни по шагам:**
> ```bash
> pwd           # Запомни где ты
> cd ..         # Перейди выше
> pwd           # Где оказался?
> cd ~          # Вернись домой
> pwd           # Дома?
> ls .          # Что в домашней папке?
> ```
> После каждой команды — пауза. Подумай: что именно изменилось?

## Почему пробелы в YAML критичны

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/yaml_indentation.png" alt="YAML отступы" width="90%"/>
<br/><em>Невидимые пробелы — видимый результат. Один лишний пробел = ошибка запуска.</em>
</div>

Когда дойдём до Docker Compose и Kubernetes, ты будешь писать файлы в формате **YAML**. В нём структура определяется **пробелами в начале строки** — как вложенность папок в Проводнике.

```yaml
# ✅ ПРАВИЛЬНО — ровно 2 пробела на каждый уровень вложенности
services:
  web:            # ← 2 пробела
    image: nginx  # ← 4 пробела (вложено в web)
    ports:        # ← 4 пробела
      - "80:80"   # ← 6 пробелов (вложено в ports)

# ❌ НЕПРАВИЛЬНО — 3 пробела вместо 2
services:
   web:           # ← 3 пробела = ошибка!
```

**Три правила YAML которые спасут от боли:**
1. Только **пробелы**, никогда Tab
2. Всегда **2 пробела** на один уровень вложенности
3. Одинаковые уровни = одинаковое количество пробелов

> **Настрой редактор один раз:** VS Code → Settings → «Tab Size» = 2, «Insert Spaces» = true. После этого клавиша Tab автоматически вставляет 2 пробела.

## Что происходит когда пишешь git add и git commit

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/git_stages.png" alt="Три зоны Git" width="90%"/>
<br/><em>Черновики на столе → стопка для сдачи → архив. Git работает так же.</em>
</div>

Это забегаем вперёд — подробно разберём в модуле 2. Но знать эту картинку заранее полезно.

У Git три зоны:

| Зона | Что это | Как туда попасть |
|------|---------|-----------------|
| **Рабочая папка** | Файлы которые ты редактируешь прямо сейчас | Просто редактируй файлы |
| **Staging area** | «Корзина для сохранения» — что войдёт в следующий снимок | `git add имя_файла` или `git add .` |
| **Repository** | Архив снимков с датой и описанием | `git commit -m "описание"` |

**Зачем нужна промежуточная зона?** Ты изменил 10 файлов, но хочешь сохранить только 3 — те, которые относятся к одной задаче. `git add` позволяет выбрать именно эти 3.

## Как строится Docker-образ слой за слоем

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/audit-unified-illustrations-faq/images/docker_build_layers.png" alt="Слои Docker образа" width="90%"/>
<br/><em>Каждая строка Dockerfile — один слой. Изменилась строка — пересобирается слой и все после него.</em>
</div>

Тоже забегаем вперёд — разберём в модуле 3. Но картинку запомни сейчас.

```dockerfile
FROM python:3.11-slim     # Слой 1: базовый образ (скачивается из интернета)
RUN pip install flask     # Слой 2: установить Flask (долго — только при первом запуске!)
COPY app.py /app/         # Слой 3: скопировать твой код (быстро)
CMD ["python", "app.py"]  # Слой 4: что запускать
```

**Хитрость Docker:** если строка не изменилась — берёт слой из кэша мгновенно. Поэтому зависимости (`pip install`) ставим **до** копирования кода — они меняются редко, а код — часто.

## ❓ Частые вопросы новичков по этому уроку

<details>
<summary><b>Что такое «сервер»? Это просто мощный компьютер?</b></summary>

Да, сервер — это компьютер. Но его задача — **работать для других**, а не для тебя лично.

- Сервер работает **24 часа в сутки, 7 дней в неделю**
- К нему подключаются **тысячи людей одновременно**
- У сервера **нет экрана, мышки, клавиатуры** — только сеть и питание
- Серверы стоят в **дата-центрах** — специальных холодных зданиях с тысячами железных шкафов

> 🏭 Аналогия: твой компьютер — маленькая домашняя кухня. Сервер — промышленная кухня ресторана, которая работает круглосуточно и кормит 500 человек.

</details>

<details>
<summary><b>Три слоя приложения — какие именно три?</b></summary>

Любое приложение (сайт, игра, банк) состоит из трёх слоёв:

1. **Frontend** — то, что видишь ты: кнопки, картинки, текст
2. **Backend** — невидимая часть, обрабатывающая действия
3. **Database** — хранилище всех данных

> 🍕 Аналогия: Frontend = зал с меню, Backend = повар на кухне, Database = кладовая с продуктами.

</details>

<details>
<summary><b>Что такое «переменная окружения»? Это из математики?</b></summary>

Нет! «Переменная» = «настройка, которую можно менять», «окружение» = контекст запуска программы.

```bash
DB_PASSWORD=МойСекрет123   # не пиши пароли прямо в коде!
APP_PORT=8080
```

Зачем? Если написать пароль прямо в коде и выложить на GitHub — его увидит **весь мир**. Переменная окружения хранится отдельно и в репозиторий не попадает.

</details>
