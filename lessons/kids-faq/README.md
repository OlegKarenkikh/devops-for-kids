## 📦 Модуль 0 — Основы

### ❓ Что такое «сервер»? Это просто мощный компьютер?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_server.jpg" alt="Сервер vs обычный компьютер" width="90%"/>
<br/><em>Главная разница — не мощность, а назначение: сервер работает для других, а не для тебя</em>
</div>

Да, сервер — это компьютер. Но у него другая задача: **работать для других людей**, а не для тебя лично.

- Твой компьютер включается когда ты хочешь → сервер работает **24 часа 7 дней в неделю**
- Ты один работаешь за компьютером → к серверу подключаются **тысячи людей одновременно**
- У компьютера есть экран, мышка, клавиатура → у сервера **ничего этого нет**, только провода
- Твой компьютер стоит дома → серверы стоят в специальных зданиях — **дата-центрах** (там холодно, темно и много-много железных шкафов)

> 🏭 Аналогия: твой компьютер — это маленькая домашняя кухня. Сервер — это промышленная кухня ресторана, которая работает всю ночь и кормит 500 человек.

### ❓ Три слоя приложения — какие именно три?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_three_layers.jpg" alt="Три слоя приложения" width="90%"/>
<br/><em>Любое приложение в мире — от ВКонтакте до банка — состоит из этих трёх слоёв</em>
</div>

Любое приложение (сайт, игра, банковское приложение) устроено одинаково — три слоя:

1. **Frontend (Интерфейс)** — то, что видишь ты на экране: кнопки, картинки, текст  
   *Пример: страница ВКонтакте в браузере*

2. **Backend (Логика)** — невидимая часть, которая обрабатывает твои действия  
   *Пример: когда ты нажимаешь «лайк» — backend сохраняет это в базу данных*

3. **Database (База данных)** — хранилище всех данных  
   *Пример: все посты, фото, друзья — всё хранится здесь*

> 🍕 Аналогия с рестораном: Frontend = зал с меню (ты видишь), Backend = повар на кухне (обрабатывает заказ), Database = кладовая с продуктами (хранит всё нужное).

### ❓ Что такое «порт»? Это как порт для кораблей?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_ports.jpg" alt="Порты компьютера как квартиры" width="90%"/>
<br/><em>Один IP-адрес — тысячи портов. Каждый порт — отдельный «вход» для разных программ</em>
</div>

Отличная аналогия! Порт в компьютере **очень похож** на порт для кораблей.

Твой компьютер имеет один IP-адрес (как адрес дома). Но на нём работают много программ одновременно. Порт — это **номер «квартиры»** в этом доме.

| Порт | Что там живёт |
|------|--------------|
| :80  | Обычные сайты (HTTP) |
| :443 | Защищённые сайты (HTTPS) |
| :22  | SSH — удалённый вход в сервер |
| :5432 | PostgreSQL — база данных |
| :8080 | Твоё приложение (по умолчанию) |
| :3000 | Node.js / React приложения |

```bash
# Посмотреть, какие порты открыты на твоём компьютере:
ss -tulpn
# или
netstat -tulpn
```

> 📮 Аналогия: IP-адрес = адрес дома, порт = номер квартиры. Письмо (запрос) приходит на адрес дома, а потом доставляется в конкретную квартиру.

### ❓ «Переменная окружения» — это из математики?

Нет, ничего общего с математикой! Слово «переменная» здесь значит «настройка которую можно менять».

«Окружение» (environment) — это контекст, в котором запускается программа.

```bash
# Переменная окружения — просто имя = значение:
DB_PASSWORD=МойПароль123
APP_PORT=8080
DEBUG=false

# Программа читает её так (Python):
import os
password = os.environ["DB_PASSWORD"]   # Читаем переменную
```

Зачем это нужно? Чтобы **не писать секреты в код**. Если ты пишешь пароль прямо в программе и выкладываешь на GitHub — его видит весь мир. А переменная окружения хранится отдельно и в GitHub не попадает.

### ❓ Почему команды такие короткие (ls, cd, pwd)?

Потому что их придумали в **1970-х годах**, когда клавиатуры были медленными и дорогими, а интернет передавал данные очень медленно. Чем короче команда — тем быстрее набрать и передать.

| Короткая | Расшифровка | Что делает |
|---------|------------|-----------|
| `ls` | list | Показать список файлов |
| `cd` | change directory | Сменить папку |
| `pwd` | print working directory | Показать текущую папку |
| `rm` | remove | Удалить |
| `cp` | copy | Копировать |
| `mv` | move | Переместить/переименовать |
| `mkdir` | make directory | Создать папку |

Со временем ты запомнишь их автоматически — как буквы на клавиатуре.

### ❓ Что такое права доступа и chmod 755?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_chmod.jpg" alt="chmod 755 расшифровка" width="90%"/>
<br/><em>Три цифры = три группы людей. Каждая цифра = сумма прав: 4=читать, 2=писать, 1=запускать</em>
</div>

Права доступа — это система **«кому что разрешено»** делать с файлом.

Три группы людей:
1. **Владелец** (owner) — тот, кто создал файл
2. **Группа** (group) — коллеги из одной группы
3. **Все остальные** (others) — все остальные пользователи

Три вида прав:
- **r** (read = 4) — читать файл
- **w** (write = 2) — изменять файл  
- **x** (execute = 1) — запускать как программу

**Число = сумма прав:**
- 7 = 4+2+1 = rwx (читать + писать + запускать)
- 5 = 4+0+1 = r-x (читать + запускать, но не писать)
- 4 = 4+0+0 = r-- (только читать)
- 0 = нет прав

```bash
chmod 755 script.sh   # Владелец: всё. Группа: читать+запускать. Все: читать+запускать
chmod 644 file.txt    # Владелец: читать+писать. Все: только читать
chmod +x script.sh    # Просто добавить право запускать
```

### ❓ Знак $ в начале строки — это часть команды?

**Нет!** Знак `$` — это приглашение терминала (prompt). Он просто показывает, что терминал ждёт твою команду.

```bash
$ ls          ← знак $ НЕ набираешь, это просто обозначение в документации
$ cd /home    ← вводишь только: cd /home
```

Когда видишь в инструкции `$ команда` — набирай только то, что после `$`.

> Если в начале строки стоит `#` — это **команда для root** (администратора). Будь осторожен!

---

### ❓ Что такое root и зачем он нужен?

Root — это **главный администратор** системы Linux. Он может всё: удалить любой файл, сломать систему, установить любую программу.

```bash
whoami        # Узнать кто ты
sudo whoami   # Выполнить как root (покажет: root)
```

> ⚠️ Правило: не работай под root постоянно. Одна опечатка и можно удалить всю систему. Используй `sudo` только когда действительно нужно.

---

### ❓ Зачем нужен bash-скрипт?

Скрипт — это **список команд**, записанный в файл, чтобы выполнить их все сразу.

Пример: ты каждый день делаешь одно и то же:
```bash
# Без скрипта — каждый раз вручную:
git pull
npm install
npm run build
docker build -t myapp .
docker push myapp

# Со скриптом — одна команда:
./deploy.sh
```

Скрипт — это как макрос в Excel или автоматизация в телефоне: один раз написал, потом запускаешь одной командой.

### ❓ Git и GitHub — это одно и то же?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_git_github.jpg" alt="Git vs GitHub — в чём разница" width="90%"/>
<br/><em>Git = программа на твоём компьютере. GitHub = сайт в интернете. Разные вещи!</em>
</div>

Нет, это **разные вещи**:

| | Git | GitHub |
|---|---|---|
| Что это | Программа на компьютере | Сайт в интернете |
| Где хранит | На твоём диске | В облаке (серверы Microsoft) |
| Кто видит | Только ты | Можно сделать публичным |
| Без интернета | Работает | Не работает |

**Аналогия:** Git — это твой личный дневник. GitHub — это публичная библиотека, где ты выкладываешь копию своего дневника, чтобы другие могли читать.

Существуют и другие аналоги GitHub: **GitLab** (популярен в России), **Bitbucket**, **Gitea** (самоходный).

---

### ❓ Что такое «ветка» в Git?

Ветка — это **параллельная копия** твоего кода, в которой можно экспериментировать, не ломая основной код.

```bash
git branch           # Посмотреть текущие ветки
git checkout -b feature/новая-кнопка   # Создать новую ветку
# ... делаешь изменения ...
git add .
git commit -m "Добавил новую кнопку"
git checkout main    # Вернуться на основную ветку
git merge feature/новая-кнопка  # Влить изменения
```

> 🎮 Аналогия: ветка — как «сохранить и попробовать другой путь» в RPG-игре. Если эксперимент не удался — просто удаляешь ветку и всё остаётся как было.

---

### ❓ SSH-ключ — это как замок с ключом?

Точно! SSH-ключ работает **именно как замок с ключом**.

- **Публичный ключ** (`.pub`) = замок — ты даёшь его серверу/GitHub
- **Приватный ключ** = ключ — хранится только у тебя, никому не показываешь

```bash
# Создать пару ключей:
ssh-keygen -t ed25519 -C "твой@email.com"
# Появятся два файла:
# ~/.ssh/id_ed25519      ← приватный (только у тебя!)
# ~/.ssh/id_ed25519.pub  ← публичный (даёшь серверам)

# Скопировать публичный ключ:
cat ~/.ssh/id_ed25519.pub
# Этот текст вставляешь в GitHub: Settings → SSH Keys
```

> 🔐 Правило: приватный ключ как паспорт — никому не показывай и никуда не копируй в открытом виде.

---

### ❓ git push не принимает пароль — что делать?

GitHub убрал поддержку паролей в августе 2021. Теперь нужен **Personal Access Token (PAT)** или SSH-ключ.

```bash
# Вариант 1 — PAT (быстро):
# Создать: github.com → Settings → Developer Settings → Personal Access Tokens
git config --global credential.helper store
git push
# При первом запросе введи:
#   Username: твой_логин_github
#   Password: ghp_твой_токен  ← именно токен, не пароль аккаунта!
# Токен сохранится в ~/.git-credentials

# Вариант 2 — SSH (один раз, навсегда):
# → Модуль 2, Урок 11
```

| | PAT | SSH |
|---|---|---|
| Настройка | 2 минуты | 5 минут |
| Срок жизни | Ограничен (ставь 90 дней) | Бессрочно |
| Безопасность | Средняя | Высокая |
| Рекомендуется для | Скриптов и автоматизации | Ежедневной работы |

---

### ❓ curl — это что? Браузер в терминале?

Почти! `curl` — инструмент для отправки HTTP-запросов из терминала. Как браузер, но без окон и кнопок.

```bash
# Простые примеры:
curl https://api.github.com          # GET-запрос (как открыть страницу)
curl -I https://google.com           # Только заголовки ответа

# С токеном (GitHub API):
curl -H "Authorization: Bearer $GITHUB_TOKEN" https://api.github.com/user

# POST-запрос (отправить данные):
curl -X POST \
  -H "Content-Type: application/json" \
  -d '{"name":"test"}' \
  https://api.github.com/user/repos
```

DevOps используют `curl` для автоматизации — деплой, проверка здоровья сервиса, работа с API — всё без браузера.

---

### ❓ PAT vs SSH — когда что использовать?

| Ситуация | PAT | SSH |
|----------|-----|-----|
| Скрипт автоматизации (CI/CD) | ✅ Предпочтительно | ⚠️ Сложнее настроить |
| Ежедневный git push/pull | ⚠️ Нужно обновлять | ✅ Предпочтительно |
| curl к GitHub API | ✅ Да | ❌ Не применимо |
| Shared-сервер (не твой) | ✅ Создай отдельный ограниченный токен | ⚠️ Осторожно |

> 🔐 **Правило:** токен как пароль — не коммить в код, не оставлять в `~/.bash_history`. Отзывай после использования.

---

### ❓ Зачем .gitignore?

.gitignore — файл со списком того, что Git должен **игнорировать** (не отслеживать).

Зачем? Некоторые файлы **не должны** попадать в GitHub:
- `.env` — пароли и секреты
- `node_modules/` — тысячи файлов библиотек (их можно скачать заново)
- `__pycache__/` — скомпилированный Python-кэш
- `.DS_Store` — служебные файлы macOS

```bash
# .gitignore — просто текстовый файл:
.env
*.log
node_modules/
__pycache__/
.DS_Store
*.pyc
```

### ❓ В чём разница между Image и Container?

<div align="center">

| | Docker Image | Docker Container |
|---|---|---|
| Что это | Шаблон/снимок | Живая копия, работающая сейчас |
| Можно изменить | ❌ Нет | ✅ Да (но изменения пропадут при удалении) |

```bash

docker ps               # Посмотреть запущенные контейнеры
docker ps -a            # Все контейнеры (включая остановленные)
docker run nginx        # Запустить контейнер из Image "nginx"
```

### ❓ Порт 8080:8080 — почему два раза?

Потому что это **два разных мира** — твой компьютер и контейнер.

```bash
docker run -p 8080:8080 моё-приложение
              ↑     ↑
              |     Порт ВНУТРИ контейнера (где слушает приложение)
              Порт на ТВОЁМ компьютере (куда ты обращаешься)
```

Их можно сделать разными:
```bash
docker run -p 9999:8080 приложение
# Твоё приложение слушает на 8080 внутри
# Ты открываешь http://localhost:9999 на своём компьютере
```

> 🚪 Аналогия: контейнер — отдельная квартира. Ты стучишь в дверь 9999 своего дома, а система перенаправляет тебя в дверь 8080 квартиры-контейнера.

---

### ❓ Зачем нужен Volume?

Volume — это способ **сохранить данные**, когда контейнер удаляется.

Проблема: контейнер — временный. Когда ты его удаляешь — все данные внутри пропадают. Если там была база данных с записями пользователей — они все потеряются!

```yaml
# В docker-compose.yml:
services:
  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data  # Данные хранятся в volume "pgdata"

volumes:
  pgdata:   # Объявляем volume — Docker создаёт папку на хосте
```

> 📦 Аналогия: контейнер — арендованная квартира. Volume — личный склад, куда ты перевозишь вещи перед тем как съехать. Квартира меняется, вещи остаются.

## 🐙 Модуль 4 — Docker Compose

### ❓ Зачем Compose если есть docker run?

`docker run` запускает **один** контейнер. Но реальное приложение — это обычно несколько:
- веб-сервер (nginx)
- приложение (Flask/Node)
- база данных (PostgreSQL)
- кэш (Redis)

Запускать каждый вручную — мучение. Compose запускает **всё одной командой**:

```bash
docker compose up -d      # Поднять весь стек
docker compose down       # Остановить всё
docker compose logs -f    # Логи всего стека
docker compose ps         # Статус всех сервисов
```

> 🎸 Аналогия: `docker run` — это один музыкант. `docker compose` — дирижёр, который поднимает сразу весь оркестр одной командой.

---

### ❓ Что значит `depends_on` в docker-compose.yml?

`depends_on` говорит Compose: «запусти этот сервис ПОСЛЕ того как запустится другой».

```yaml
services:
  web:
    image: myapp
    depends_on:
      - db        # web стартует только после db
  db:
    image: postgres
```

⚠️ Важно: `depends_on` ждёт только старта контейнера, но **не готовности** базы данных. Для полной готовности нужен `healthcheck`.

---

### ❓ Зачем нужны сети (networks) в Compose?

По умолчанию все сервисы в одном compose-файле видят друг друга по имени. Но сети позволяют **изолировать** сервисы:

```yaml
networks:
  frontend:     # видят: web и nginx
  backend:      # видят: web и db (db не доступен снаружи!)
```

> 🏢 Аналогия: в офисе разные отделы работают в разных комнатах. Бухгалтерия не имеет прямого доступа к серверной.

---

### ❓ .env и docker-compose.yml — как они связаны?

Compose автоматически читает файл `.env` из той же папки:

```yaml
# docker-compose.yml
services:
  db:
    image: postgres
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # берётся из .env
      POSTGRES_DB: ${DB_NAME}
```

```bash
# .env (не коммитить в Git!)
DB_PASSWORD=supersecret
DB_NAME=myapp
```

Так пароли хранятся в `.env`, а не в коде. `.env` добавляется в `.gitignore`.

### ❓ «Kubernetes» — откуда это слово?

Kubernetes — это **греческое слово** (κυβερνήτης), означающее «кормчий» или «рулевой». Тот, кто управляет кораблём.

Произносится: **«кубернэтес»** или просто **«кубик»** (k8s — буква k, потом 8 букв, потом s).

```text
K   u   b   e   r   n   e   t   e   s
k   ← 8 букв →                      s
= k8s
```

Создан в Google в 2014 году, открыт как open source.

### ❓ Pod — это стручок гороха?

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_pod.jpg" alt="Pod в Kubernetes — как стручок гороха" width="90%"/>
<br/><em>Pod = стручок. Внутри — один или несколько контейнеров, которые всегда запускаются вместе</em>
</div>

Да! Слово «Pod» по-английски означает «стручок» — как стручок гороха, где горошины (контейнеры) вместе.

**Pod** — это минимальная единица в Kubernetes. Внутри одного Pod могут быть несколько контейнеров, которые:
- Всегда запускаются на одном сервере
- Видят друг друга через localhost
- Делят одну сеть и хранилище

```bash
kubectl get pods          # Список всех Pod
kubectl describe pod имя  # Подробности о Pod
kubectl logs имя          # Логи Pod
kubectl exec -it имя -- bash  # Зайти внутрь
```

### ❓ Чем Deployment отличается от Pod?

| | Pod | Deployment |
|---|---|---|
| Что делает | Запускает контейнеры | Управляет группой Pod |
| Если упадёт | Не перезапустится | Автоматически создаст новый |
| Масштаб | 1 штука | Можно указать сколько копий |
| Обновление | Вручную | Rolling update автоматически |

```yaml
# Deployment — умный менеджер для Pod:
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3        # Держать 3 копии Pod всегда запущенными
  template:
    spec:
      containers:
      - name: web
        image: nginx
```

> 👔 Аналогия: Pod — рабочий. Deployment — менеджер, который следит чтобы рабочих всегда было нужное количество. Один заболел — менеджер нанимает нового.

### ❓ kubectl — как произносится?

**«кубе-контрол»** (cube-control) или просто **«кубектл»**. В английском сообществе говорят и так и так — единого стандарта нет.

`kubectl` = **kube** (Kubernetes) + **ctl** (control)

## 🔐 Модуль 6 — Секреты и API

### ❓ .env — точка в начале что означает?

В Linux точка в начале имени файла означает **скрытый файл**. Его не видно в обычном `ls`, нужно `ls -a` (all).

```bash
ls          # Не покажет .env
ls -a       # Покажет все файлы включая скрытые
ls -la      # Подробно показать все файлы
```

Зачем скрывать? Чтобы случайно не выложить на GitHub и не удалить.

### ❓ REST API — REST это «отдых»?

Почти! REST расшифровывается как **RE**presentational **S**tate **T**ransfer — «передача состояния представления». Звучит страшно, но суть простая.

REST — это **правила** как строить API (интерфейс для программ):

<div align="center">
<img src="https://raw.githubusercontent.com/OlegKarenkikh/devops-for-kids/main/images/kid_rest_api.jpg" alt="REST API методы GET POST PUT DELETE" width="90%"/>
<br/><em>4 метода HTTP = 4 действия: получить, создать, обновить, удалить</em>
</div>

| HTTP метод | Действие | Пример |
|-----------|---------|--------|
| **GET** | Получить данные | `GET /users` — список пользователей |
| **POST** | Создать новое | `POST /users` — создать пользователя |
| **PUT** | Обновить полностью | `PUT /users/1` — обновить пользователя |
| **DELETE** | Удалить | `DELETE /users/1` — удалить пользователя |

```bash
# Пример с curl:
curl http://localhost:8080/items          # GET — получить все
curl -X POST http://localhost:8080/items  # POST — создать
```

### ❓ JSON — это имя человека?

JSON расшифровывается как **J**ava**S**cript **O**bject **N**otation — формат записи данных, придуманный для JavaScript.

Нет, не имя — аббревиатура! Придумал Дуглас Крокфорд (Douglas Crockford) около 2001 года.

```json
{
  "имя": "Вася",
  "возраст": 12,
  "любимые_игры": ["Minecraft", "Roblox"],
  "адрес": {
    "город": "Москва",
    "улица": "Ленина"
  }
}
```

JSON — универсальный язык данных. Программы на Python, Java, Go, JavaScript — все понимают JSON. Это как «эсперанто» для данных.

## 🎓 Бонусный модуль

### ❓ OSI — что это за буквы?

OSI = **O**pen **S**ystems **I**nterconnection (Взаимодействие открытых систем). Придумана в 1984 году международной организацией ISO, чтобы объяснить как устроены сети.

7 уровней нужны потому, что каждый делает одну вещь хорошо — **разделение ответственности**:
- Физика отвечает за провода
- Сеть — за маршрутизацию
- Транспорт — за доставку
- Приложение — за данные

Это как на заводе: один рабочий точит детали, другой собирает, третий красит — каждый специалист в своём деле.

### ❓ HTTPS — буква S означает «секретный»?

Почти! S = **S**ecure (безопасный). HTTPS = HTTP + шифрование (TLS).

- **HTTP** — данные передаются открытым текстом (любой может прочитать)
- **HTTPS** — данные зашифрованы (можно прочитать только получателю)

TLS (Transport Layer Security) и SSL (Secure Sockets Layer) — это названия протоколов шифрования. SSL — старое название, TLS — новое, но люди часто говорят «SSL-сертификат» имея в виду TLS.

```bash
# Проверить сертификат сайта:
curl -I https://google.com | grep -i "HTTP"
openssl s_client -connect google.com:443
```

### ❓ LLM — почему «языковая» модель?

LLM = **L**arge **L**anguage **M**odel (Большая Языковая Модель).

«Языковая» — потому что она обучена на **текстах** (языке), а не на картинках или звуках. Она научилась предсказывать следующее слово в тексте, прочитав почти весь интернет.

**GPU для ИИ** быстрее CPU потому что:
- CPU — несколько мощных ядер (4-32), делает сложные задачи по очереди
- GPU — тысячи маленьких ядер (тысячи!), делает простые задачи параллельно

Нейросети — это миллиарды умножений матриц. GPU делает их все одновременно, CPU — по одному.
